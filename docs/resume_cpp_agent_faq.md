# 常见问题与回答（基于本项目实现，不编造）

## 1. 这个项目整体架构是什么？
项目是一个 C++20 的 HTTP 服务：负责鉴权、路由分发、SSE 流式输出与业务编排；底层通过 MySQL 持久化会话/消息/笔记/分块，通过 Qdrant 保存可检索的向量记忆；在面试业务上采用可插拔的 Agent 流水线把“检索增强 + 生成”拆成多个阶段，便于迭代。

## 2. 你说的 Agent 流水线具体是什么？每个阶段做什么？
本项目把一次面试回合拆为五阶段：
- Plan：生成本回合的基本计划信息（例如方向、是否已有回答、模式标识）
- Retrieve：从向量记忆中召回与本回合相关的 chunk（按会话与记忆类型过滤）
- Compose：把 JD、简历、召回记忆按模板组织成最终 messages，并做上下文预算控制
- Generate：调用 LLM 并以 SSE 形式流式输出（delta/done）
- Reflect：预留反思输出接口（当前默认实现返回空对象）

这样拆分的意义是：逻辑边界清晰，检索与提示词模板可以独立演进，后续替换任一阶段成本更低。

## 3. RAG 是怎么做的？“检索增强”具体增强在哪里？
项目采用“先向量化，再向量检索，再注入上下文”的方式：
1) 选择一个 query（面试开始时可用 JD，回答后可用用户回答）
2) 调用 embedding 接口得到向量
3) 在 Qdrant 中按 `session_id + memory_type` 做过滤检索，取 topK 记忆 chunk
4) 在 Compose 阶段把这些 chunk 作为“MEMORY”上下文注入到 system prompt 中，并控制字符预算，防止上下文膨胀

增强点在于：模型在生成追问/反馈时不仅依赖当前输入，还能利用历史会话沉淀的结构化记忆片段，提升一致性与针对性。

## 4. “结构化记忆索引”是怎么产生的？什么时候触发？
当某个会话的历史消息数超过阈值（可配置）时，会异步触发索引任务：
- 拉取该会话尾部一段对话作为 transcript
- 调用 LLM 生成“严格 JSON”结构化笔记（facts/preferences/todos/weaknesses/conclusions/tags/memory_type 等）
- 将笔记渲染为可读文本并分块
- 对分块批量 embedding
- 写入 Qdrant（向量点 payload 带 session_id/memory_type/note_id/chunk_index/text_hash/text 等）

之所以异步执行，是为了不阻塞 SSE 输出与请求线程。

## 5. 如果 LLM 没按要求返回严格 JSON，会怎样？
项目会尝试从输出中提取并解析 JSON；如果仍然无法解析为对象，会降级为保存原始文本，并补充必要字段，保证索引流水线不会因为格式偏差直接中断。

## 6. 为什么要做分块（chunk）？chunk 大小/批量大小有什么考虑？
分块主要是为了：
- 控制 embedding 单条输入长度，降低超长文本导致的失败概率
- 提升召回粒度：检索时返回更具体的片段，便于预算裁剪与拼接
项目实现中按“按行拼接到最大字符数”为主进行分块，并以固定 batch 大小批量调用 embedding。

## 7. SSE 流式输出是怎么设计的？客户端怎么判断结束？
项目 SSE 约定了三类事件：
- `delta`：增量文本（每次模型输出一段就发一次）
- `error`：错误信息
- `done`：结束信号（包含 done=true 与输出长度等信息）

客户端以 `done` 事件作为一次请求的结束标志。

## 8. 这个项目如何做鉴权？token 放哪里？
鉴权使用 JWT：
- 登录接口成功后通过 Cookie 写入 `token=...`
- 后续请求既支持 `Authorization: Bearer ...`，也支持从 Cookie 读取 token
- 中间件会校验 token，有效后把用户名写入请求头（用于后续 DB 隔离与审计）

## 9. 为什么调试路由默认关闭？怎么开启？
因为调试接口（例如 SSE hold）属于“验证/诊断用途”，默认暴露会扩大攻击面与误用风险。
项目默认不注册/不放行 `/debug/*`，只有显式设置环境变量开启后才会启用。

## 10. 如何避免开源时把密钥提交到仓库？
项目提供配置模板，并通过忽略规则把本地实际配置文件排除在版本控制之外：
- 读者复制 `*.template` 生成本地文件填入密钥
- 仓库默认不包含可用的明文 key/token

## 11. 配置优先级是怎样的？
项目采用“环境变量覆盖配置文件，配置文件覆盖默认值”的方式：
- 配置文件加载只是把 key/value 写入进程环境
- 读取时统一从环境变量取值，并提供默认值

## 12. 有哪些可回归验证方式？如果没有外部依赖也能跑吗？
项目提供多种层级的验证：
- 最小验证：启动服务 + /health + SSE hold（不依赖外部 LLM/Qdrant/Embedding）
- 自检二进制：覆盖 LLM profiles、SSE、embedding、qdrant、记忆索引等（部分会在未配置外部依赖时自动跳过）
- CI：默认跑构建与最小 smoketest，确保开源后每次变更可回归

## 13. 当前项目明确“还没有做”的点有哪些？
为了避免夸大，这里列出当前仓库没有体现为实现的方向（至少不作为本项目已具备能力来描述）：
- 分布式追踪/链路观测（Tracing/metrics/structured logging）
- 熔断/限流/重试策略的系统化治理（除了已有的超时配置与错误返回）
- 多租户权限模型与更细粒度的 RBAC
- 向量记忆的重排序、评估平台、在线 AB 与自动化基准报告
